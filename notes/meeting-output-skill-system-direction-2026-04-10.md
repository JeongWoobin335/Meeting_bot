# 결과물 생성 SKILL 시스템 방향 정리

작성일: 2026-04-10  
대상 워크스페이스: `C:\Users\jung\Desktop\ZOOM_MEETING_BOT`

## 1. 이 문서를 쓰는 이유

이 문서는 `ZOOM_MEETING_BOT`에서 진행 중인 `meeting-output skill` 작업의 배경, 원칙, 현재 중심 과제, 이미 진행한 작업, 앞으로의 방향을 한 번에 정리하기 위한 내부 개발 문서다.

지금까지 대화가 길어지면서 아래 같은 중요한 판단들이 여러 번 나왔다.

- 이 작업은 단순히 회의록 문체를 바꾸는 작업이 아니다.
- 이 작업은 본체 엔진을 사용자에게 개방하는 작업도 아니다.
- 이 작업의 핵심은 `결과물 생성 파트 전체에 대한 AI 지시/가이드/지침`을 코드 밖 `SKILL.md` 자산으로 분리하는 것이다.
- 사용자는 코드를 수정하는 것이 아니라, 자기 `SKILL.md` 자산을 축적하면서 결과물을 확장해 나간다.
- 사용자가 보는 자산은 딱 `SKILL.md` 하나여야 한다.
- 내부 엔진 계약, 세션/전사/전달 흐름은 절대 사용자가 제멋대로 바꿀 수 있게 하면 안 된다.

이 문서는 위 판단들을 다시 흩어지지 않게 고정하기 위한 기준 문서다.

## 2. 우리가 처음부터 만들고 싶었던 것

처음부터 의도한 것은 다음과 같다.

- 사용자가 `ZOOM_MEETING_BOT`를 실행하면 본체 엔진은 안정적으로 회의에 들어간다.
- 본체는 회의 참가, 수집, 전사, 화자 파악, 요약 생성, PDF 생성, 전달 같은 핵심 흐름을 책임진다.
- 하지만 `결과물을 어떤 방식으로 만들지`에 대한 세부 지침은 코드에 박혀 있지 않고 `SKILL.md`로 분리되어 있어야 한다.
- 그래서 사용자는 개발자에게 매번 “이 양식으로 바꿔주세요”, “이 회의록은 이런 느낌으로 써주세요”라고 요청하지 않아도 된다.
- 사용자는 자기 취향, 자기 회사, 자기 회의 유형, 자기 업무 방식에 맞는 결과물 생성 전략을 자기 `SKILL.md` 자산으로 계속 쌓아갈 수 있다.

즉 요약하면, 본체 엔진은 개발자가 관리하고, 결과물 생성 레이어는 사용자가 자기 자산으로 확장하는 구조를 만들고 싶었던 것이다.

## 3. 왜 이 작업이 필요한가

이 작업은 단순한 커스터마이징 편의 기능이 아니다. 시스템의 성장 방식 자체와 관련 있다.

- 지금까지는 개발자가 만든 코드 안에 결과물 생성 지침이 들어 있어, 결국 사용자는 개발자가 만든 100% 한도 안에서만 사용할 수 있다.
- 하지만 결과물 생성 지침을 `SKILL.md`로 분리하면, 사용자는 자기 능력과 감각으로 그 위를 확장할 수 있다.
- 즉 시스템의 공통 엔진은 개발자의 100%를 유지하면서, 사용자마다 자기 퍼센트를 더해 100% 이상으로 발전시킬 수 있게 된다.

이 구조가 중요한 이유는 아래와 같다.

- 개발자는 본체 엔진의 품질과 안정성에 집중할 수 있다.
- 사용자는 코드 수정 없이 자기 문서 스타일과 결과물 방식을 자산화할 수 있다.
- 결과물 생성 전략이 사람마다 달라져도 엔진은 하나로 유지된다.
- 사용자 자산이 파일로 남기 때문에 재사용, 복사, 변형, 축적이 가능하다.

## 4. 이 작업에서 절대 흔들리면 안 되는 원칙

### 4-1. 본체 엔진은 유지한다

사용자가 바꾸면 안 되는 것은 분명하다.

- Zoom 참가 방식
- 음성/채팅 수집
- 전사 파이프라인
- 화자 추론 파이프라인
- 세션 관리
- 큐와 finalization
- PDF 생성/전달 흐름의 엔진 계약
- 최소한의 결과물 생성 보장
- 내부 JSON schema

이런 것까지 skill로 넘기면, 더 이상 `ZOOM_MEETING_BOT`라는 공통 시스템의 의미가 없다. 그 순간 사용자는 본체를 확장하는 것이 아니라 본체를 해체하는 셈이 된다.

### 4-2. `0의 상태`는 필요하지만 감옥이면 안 된다

기본 결과물 레퍼런스는 필요하다.

- 아무 skill이 없어도 AI가 엉뚱한 결과물을 만들지 않게 해주는 기준본이 필요하다.
- 즉 `0의 상태`는 안정된 기본 결과물 레퍼런스여야 한다.

하지만 `0의 상태`가 사용자를 가두는 상한선이 되면 안 된다.

- `메모는 절대 못 뺀다`
- `결정사항은 이름만 바뀌고 의미는 못 바꾼다`
- `브랜드 느낌은 살짝만 힌트로 반영된다`

이런 식으로 0의 상태가 너무 강해지면 skill의 존재 이유가 약해진다.

정확한 원칙은 이것이다.

- `0의 상태`는 시작점이어야 한다.
- 사용자는 skill을 통해 그 0을 자기 입맛과 회의 성격에 맞게 더 발전시킬 수 있어야 한다.

### 4-3. 사용자에게 내부 구조를 설명하지 않는다

사용자는 내부 구현을 알 필요가 없다.

사용자가 몰라도 되는 것:

- metadata
- renderer_profile
- internal slot
- JSON key
- decisions slot
- postprocess_requests
- tool bridge

사용자가 해야 하는 것은 오직 원하는 결과를 말하는 것이다.

예를 들면 사용자는 이렇게 말해야 한다.

- “메모는 빼줘”
- “열린 질문 말고 검토사항처럼 보여줘”
- “카카오 느낌으로 만들어줘”
- “전체 요약을 더 풍부하게”
- “이미지 두 장 넣어줘”

시스템은 그 말을 듣고 내부적으로 해석하고 skill에 반영해야 한다.

즉 사용자 경험은 `말로 설명 -> SKILL.md 자산 생성 -> 결과물 반영`이어야 한다.

### 4-4. 사용자 자산은 `SKILL.md` 하나다

사용자에게 남는 자산은 `SKILL.md` 하나여야 한다.

- 사용자 관점에서 부속 JSON 파일이 보이면 안 된다.
- 보조 md 파일이 자산처럼 남으면 안 된다.
- sidecar metadata 파일이 생기면 안 된다.

내부에서 임시 파일을 써도 된다. 하지만 사용자에게 남는 영구 자산은 오직 `SKILL.md` 하나여야 한다.

## 5. skill의 정확한 뜻

이 프로젝트에서 skill은 단순 템플릿이 아니다.

처음에 좁게 해석했던 방식:

- 회의록 문체 템플릿
- 블록 순서 오버라이드
- 일부 헤더 이름 변경

이것만으로는 부족하다.

현재 우리가 맞다고 보는 skill의 정의는 다음과 같다.

`결과물 생성 단계 전체에서 AI가 무엇을 어떻게 만들어야 하는지를 설명하는 재사용 가능한 지침 자산`

즉 skill은 아래를 포함할 수 있어야 한다.

- 요약을 무엇 중심으로 쓸지
- 어떤 블록을 보여줄지
- 어떤 블록을 숨길지
- 블록 이름을 어떻게 보여줄지
- 블록이 실제로 어떤 성격의 내용을 담아야 할지
- 문체를 어떻게 할지
- 브랜드/분위기를 어떻게 반영할지
- 이미지를 생성할지
- 추가 결과물을 무엇으로 붙일지
- 결과물 생성 이후 사용자가 다시 보고 “이렇게 고쳐야겠다”라고 판단한 내용을 새 skill로 자산화할지

즉 `회의록 스타일`이 아니라 `결과물 생성 오케스트레이션`에 더 가까운 개념이다.

## 6. 대표적인 사용자 요구 예시

다음 문장은 이 시스템이 궁극적으로 자연스럽게 받아들여야 하는 대표 요청이다.

> “음.. 나는 지금 이후에 카카오라는 회사랑 미팅을 할건데 결과물 디자인이 카카오 느낌이 나도록 디자인을 했으면 좋겠어. 그리고 메모 칸은 필요없고 열린 질문 대신 검토사항으로 바꿔줘. 그리고 회의 전체 요약과 핵심 논의 주제를 더욱 풍부하게 강조했으면 좋겠어. 그리고 요약본 작성하면서 핵심 논의 주제에 맞게 나노바나나로 관련 이미지도 2장 정도 들어가면 좋을 것 같아”

중요한 점은, 사용자는 이 말을 그냥 자연어로 하는 것이고 내부 구조는 몰라야 한다는 점이다.

시스템 내부의 바람직한 사고 방식은 아래와 같아야 한다.

### 6-1. 브랜드/무드 해석

- `카카오`가 어떤 회사인지 조사한다.
- 공개적으로 드러난 브랜드 무드, 색감, 문서 분위기를 이해한다.
- 그에 맞는 색상, 폰트, 레이아웃 감각을 결과물 생성에 반영한다.

### 6-2. 블록 의미 재정의

- `메모는 필요없다`면 과감하게 숨긴다.
- `열린 질문 대신 검토사항`이면 이름만 바꾸지 않고, 실제로 그 블록의 성격을 검토 중인 포인트 중심으로 바꾼다.

### 6-3. 서술 밀도 조정

- `회의 전체 요약`
- `핵심 논의 주제`

이 두 블록은 더 풍부하고 강하게 작성한다.

### 6-4. 이미지까지 결과물 생성의 일부로 처리

- `이미지 2장 넣기`는 사용자 입장에서 후속 요청이 아니라 최종 결과물 요구사항의 일부다.
- 따라서 시스템은 핵심 논의 주제를 보고 이미지 브리프를 만들고, 필요한 도구를 호출해 실제 이미지를 생성하고 PDF까지 반영할 수 있어야 한다.

### 6-5. 이 사고 자체를 skill 자산으로 남긴다

이 모든 결과는 결국 하나의 `SKILL.md`로 사용자 자산이 되어야 한다.

즉 사용자가 다음에 비슷한 회의를 할 때 또 처음부터 설명하지 않아도 된다.

## 7. 사용자가 말하는 `후속 처리`의 정확한 뜻

기술적으로 내부 파이프라인 안에서 이미지 생성과 삽입을 `후속 처리`라고 부를 수는 있다. 하지만 사용자 관점에서 이 표현은 핵심이 아니다.

사용자가 말하는 `후속 처리`의 정확한 의미는 아래와 같다.

- 이미 결과물이 나온 뒤 사용자가 그 결과물을 본다.
- 사용자가 “이 회의록은 이런 식으로 쓰는 게 더 좋겠다”고 말한다.
- AI는 “그럼 이 스타일을 새 skill로 정리해 저장해 둬야겠다”고 판단한다.
- 그리고 그것이 새로운 `SKILL.md` 자산이 된다.

즉 사용자에게 중요한 후속 처리는 `이미지 삽입의 기술적 단계`보다 `결과물을 보고 더 나은 생성 전략을 skill 자산으로 축적하는 것`이다.

## 8. 지금까지 실제로 진행한 작업

이 작업은 여러 단계를 거쳐 왔다.

### 8-1. 1차

- 결과물 작성 기준을 코드 밖 `SKILL.md`로 분리
- base skill / generated skill / override skill 구조 도입
- CLI에 `skill compose`, `skill status`, `skill clear`, `skill list`, `skill activate` 추가

### 8-2. 2차

- 사용자의 자연어 요청을 바탕으로 skill 초안을 생성하는 흐름 도입
- sandbox 기반 compose 흐름으로 재작업
- 코드/테스트 파일을 임의 수정하지 않도록 격리

### 8-3. 3차

- 결과물 생성 정책을 더 skill로 이동
- 블록 순서, 노출, 개수, heading, 빈 상태 문구 등을 skill metadata에서 제어
- 블록 이름만이 아니라 블록 의미까지 재정의할 수 있게 구조 확대

### 8-4. 4차

- renderer profile, theme, color, font 방향을 skill에서 제어
- 결과물 생성 과정에서 추가 시각 자료 요청을 다룰 수 있게 확장
- 이미지 파일이 준비되어 있을 경우 DOCX/PDF에 실제 삽입되도록 exporter 확장

### 8-5. 최근 단계

- 결과물 생성 materialize 단계 도입
- 브랜드 theme 해석 로직 추가
- `image_brief` 요청이 실제 이미지 생성으로 이어질 수 있는 실행 고리 추가
- 생성된 이미지 경로를 결과물에 다시 넣어 최종 PDF까지 연결

## 9. 현재 가장 중심으로 둬야 하는 작업

지금 가장 중심으로 둬야 하는 것은 아래 한 문장으로 정리된다.

`결과물 생성 단계 전체를 skill로 열어 주되, 엔진 계약은 고정하고, 사용자는 자연어로 요청한 결과가 하나의 SKILL.md 자산으로 쌓이게 만드는 것`

이 문장을 조금 더 풀면 다음과 같다.

- 사용자는 기술적인 내부 구조를 몰라야 한다.
- 사용자는 그냥 원하는 결과를 말해야 한다.
- 시스템은 그 말을 해석해 result-generation 전략으로 바꿔야 한다.
- 그 전략은 `SKILL.md` 하나의 자산으로 남아야 한다.
- 이후 회의에서는 그 skill 자산이 실제 결과물 생성에 반영되어야 한다.
- 엔진 계약과 본체 존재 의의는 흔들리면 안 된다.

## 10. 지금 이후 구현 방향

앞으로의 구현 방향은 아래 순서가 맞다.

### 10-1. 자연어에서 result-generation strategy로

사용자 자연어 요청을 아래 층으로 번역해야 한다.

- 블록 의미 재정의
- 강조 블록 선택
- 디자인/브랜드 방향 해석
- 이미지/추가 산출물 필요 판단
- 최종 문서 구조 재구성

### 10-2. result-generation strategy에서 `SKILL.md`로

위 전략은 최종적으로 하나의 `SKILL.md` 자산으로 저장되어야 한다.

사용자는 다음을 몰라도 된다.

- metadata key
- renderer_profile
- postprocess_requests
- internal slot
- decisions / open_questions 같은 내부 key

시스템이 그것들을 내부에서 skill 표현으로 바꿔야 한다.

### 10-3. `SKILL.md`에서 실제 결과물 생성으로

active skill은 아래에 실제 영향을 줘야 한다.

- 요약 프롬프트
- executive summary
- sections
- block semantics
- block order
- block visibility
- renderer theme
- 추가 이미지/시각 자료 생성
- PDF 최종 스타일

### 10-4. 결과물을 본 뒤 다시 skill 자산으로 축적

사용자가 결과물을 보고 다시 말하면 그 또한 새 skill 자산으로 남겨야 한다.

예를 들면 이런 흐름이다.

- “이번에는 너무 딱딱했어”
- “다음에는 액션 아이템을 맨 위에 두자”
- “회의 전체 요약을 더 짧고 강하게 하자”
- “공공기관 문서 같은 분위기로 바꾸자”

이런 피드백이 다시 `SKILL.md`로 자산화되어야 한다.

## 11. 지금 구현에서 지켜야 할 금지선

아래는 계속 금지선으로 유지해야 한다.

- 엔진 계약을 skill로 빼는 것
- 세션/전사/전달 파이프라인을 사용자가 바꾸게 하는 것
- 사용자에게 sidecar JSON이나 보조 md 자산을 남기는 것
- 내부 용어를 사용자 UX에 드러내는 것
- `0의 상태`를 절대 고정 기준으로 만들어 사용자를 가두는 것
- 임시방편으로만 동작하는 구조를 계속 누적하는 것

## 12. 최종 요약

이 작업의 본질은 다음과 같다.

- 본체는 개발자의 것
- 결과물 생성 레이어는 사용자 확장 영역
- 사용자는 코드를 고치지 않는다
- 사용자는 자연어로 원하는 결과를 말한다
- 시스템은 그것을 하나의 `SKILL.md` 자산으로 남긴다
- 그 skill 자산은 이후 회의 결과물 생성에 실제로 반영된다
- `0의 상태`는 안전한 시작점이지 감옥이 아니다

즉 이 시스템은 `개발자가 만든 안정적인 엔진`과 `사용자가 쌓아가는 skill 자산`이 함께 자라는 구조로 가야 한다.
