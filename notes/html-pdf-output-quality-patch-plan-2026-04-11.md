## 1. 이 문서의 목적

이 문서는 HTML/CSS 기반 PDF-first 경로를 실제 회의 결과물에 적용한 뒤 확인된 품질 문제를 정리하고, 그 문제를 해결하기 위한 추가 패치 방향을 상세하게 고정하기 위한 문서다.

이번 문서의 핵심은 두 가지다.

1. 현재 거슬리는 문제를 "느낌"이 아니라 구체적인 품질 축으로 분해한다.
2. 각 축마다 무엇이 원인이고, 어떤 파일을 어떻게 손대야 하며, 무엇이 완료 기준인지 명확히 적는다.

이 문서는 결과물 생성 레이어 품질 문서다.  
즉, 음성 수집, 전사, 참가자 추적, `제기자 / 주요 화자 / 타임스탬프` 같은 core 기능은 이 문서의 변경 대상이 아니다.

---

## 2. 절대 유지해야 할 기준

### 2-1. core 기능은 유지한다

다음은 결과물 생성 취향이나 출력 표현이 아니라 시스템 본질 기능이다.

- Zoom/오디오 수집
- 전사
- 참가자 추적
- `제기자`
- `주요 화자`
- `타임스탬프`
- 세션 축적
- briefing JSON 생성

이 문서에서 제안하는 패치는 위 기능을 건드리지 않는다.

### 2-2. 바꾸는 것은 결과물 생성 품질이다

이번 패치의 대상은 다음이다.

- PDF 표지/페이지 흐름
- HTML/CSS 인쇄 품질
- 이미지 카드 품질
- 폰트 선택 품질
- 생성 지시문 노출 여부
- 시각자료 레이아웃 다양성

### 2-3. SKILL 우선 원칙은 그대로 간다

결과물 생성 영역에서는 SKILL이 여전히 최우선 지시사항이다.

- SKILL에 있는 내용은 집행 대상이다.
- SKILL에 없는 내용만 기본 레퍼런스로 간다.
- 품질 패치는 SKILL의 자유를 줄이기 위한 것이 아니라, 그 자유를 더 자연스럽고 고급스럽게 반영하기 위한 것이다.

---

## 3. 현재 확인된 6개의 거슬림 축

이번 HTML-first 결과물 기준으로 확인된 큰 축은 다음 6가지다.

1. PDF 인쇄 잡음 및 표지 밀도 문제
2. 페이지 흐름과 과도한 여백 문제
3. 이미지 카드 잘림/프레임/경계 문제
4. 내부 생성 지시문 노출 문제
5. 추상적 폰트 지시를 실제 폰트 선택으로 연결하지 못하는 문제
6. 시각자료가 사실상 단일 템플릿 반복처럼 보이는 문제

보조 관찰:

- 액션 아이템 누락은 이번 세션에선 실제 `action_items=[]`라서 핵심 품질 문제로 보지 않는다.
- 다만 향후 액션 아이템이 존재할 때는 더 강조되게 표현해야 한다.

## 4. 축별 상세 패치 계획

## 4-1. 축 1: PDF 인쇄 잡음 및 표지 밀도 문제

### 증상

- 1페이지 상단에 날짜/문서명이 인쇄 헤더처럼 노출됨
- 하단에 `file:///...` 경로가 그대로 노출됨
- 표지 정보량에 비해 페이지가 지나치게 비어 보임
- 제목 줄바꿈이 부자연스러워 표지 완성도가 낮음

### 추정 원인

- 브라우저 PDF 출력에서 기본 인쇄 헤더/푸터가 완전히 제거되지 않았음
- 표지 영역이 `min-height` 중심으로 과하게 길게 잡혀 있음
- 제목 폭/줄바꿈/여백 제어가 현재 문서 폭에 비해 보수적으로 설정돼 있음

### 패치 방향

1. 인쇄 헤더/푸터 완전 제거
   - 브라우저 PDF 생성 옵션 재확인
   - 현재 사용 중인 headless 인쇄 옵션이 실제 Chrome/Edge 버전에서 안정적으로 적용되는지 검증
   - 필요하면 `printToPDF` 전용 경로로 변경

2. 표지 레이아웃 재설계
   - 현재 `min-height: 240mm` 구조를 고정 길이형 cover가 아니라 내용 밀도형 cover로 조정
   - `cover`를 "꽉 찬 첫 페이지"가 아니라 "적정 상단 여백 + 제목 + 개요 + 요약 preview" 구조로 재편
   - 제목/메타가 시각적으로 더 위에 밀집되도록 정렬

3. 제목 줄바꿈 규칙 개선
   - 제목 최대 폭
   - 제목 글자 크기 자동 축소
   - 너무 짧은 단어 하나가 다음 줄로 떨어지는 경우 방지

### 예상 변경 파일

- `src/local_meeting_ai_runtime/html_pdf_renderer.py`
- `doc/templates/html-meeting-output/report.css`
- 필요 시 브라우저 PDF 생성 헬퍼 모듈

### 완료 기준

- PDF 상단/하단에 날짜, 문서명, `file:///...` 경로가 노출되지 않는다.
- 1페이지가 "거의 텅 빈 표지"처럼 보이지 않는다.
- 표지 제목 줄바꿈이 부자연스럽지 않다.

## 4-2. 축 2: 페이지 흐름과 과도한 여백 문제

### 증상

- 2페이지에서 `핵심 논의 주제` 제목까지만 나오고, 실제 첫 섹션은 다음 페이지로 넘어감
- 섹션과 이미지가 큰 블록처럼 묶여 페이지 하단이 통째로 비는 경우가 있음

### 추정 원인

- `break-inside: avoid` 성격이 너무 넓게 걸려 있음
- 섹션 본문 + 이미지 카드가 하나의 무거운 덩어리로 잡힘
- 이미지 카드 크기가 커서 페이지 하단 남은 공간에 못 들어가면 브라우저가 통째로 다음 페이지로 넘김

### 패치 방향

1. 블록 분해
   - 섹션 제목/본문/trace/이미지 카드를 모두 하나로 묶지 않는다
   - 제목/본문은 먼저 흐르게 하고, 이미지만 별도 흐름으로 재배치 가능하게 바꾼다

2. 페이지 브레이크 규칙 세분화
   - 섹션 전체 `avoid` 대신
     - 제목 단독 `avoid`
     - 이미지 카드 단독 `avoid`
     - 본문은 자연 흐름 허용
   - 카드가 큰 경우 카드만 다음 페이지로 가고, 본문은 현재 페이지를 충분히 채우게 한다

3. 이미지 카드 크기 조정
   - 페이지 맥락에 따라 full-width / compact / side-by-side variant를 선택할 수 있게 한다
   - 모든 카드가 같은 높이를 강제하지 않는다

### 예상 변경 파일

- `src/local_meeting_ai_runtime/html_pdf_renderer.py`
- `doc/templates/html-meeting-output/report.css`

### 완료 기준

- `핵심 논의 주제` 제목 아래가 통째로 비어 다음 페이지로 넘어가는 현상이 크게 줄어든다.
- 페이지 하단 여백이 "내용 없음"이 아니라 "정상 여백" 수준으로 보인다.
- 섹션 흐름이 PDF에서 더 자연스럽게 이어진다.

## 4-3. 축 3: 이미지 카드 잘림/프레임/경계 문제

### 증상

- 오른쪽 hero 이미지가 잘리거나 일부가 튀어나온 느낌이 남
- 카드 경계선이 약해 문서 안에 자연스럽게 녹지 않음
- 카드 전체가 보고서 시각자료보다 기본 카드 템플릿처럼 보임
- 현재 결과물은 "프로토타입 카드" 느낌이 강하고, 목표로 삼아야 할 보고서형 AI 인포그래픽 품질과 격차가 큼

### 추정 원인

- 현재 이미지 합성 단계에서 `ImageOps.fit(...)`을 사용해 hero 박스에 맞추고 있음
- 비율이 다른 이미지를 강제로 채우면서 crop이 발생함
- 카드 경계/패널/보더/내부 간격이 정교하게 설계되지 않았음

### 패치 방향

1. crop 기반 배치 제거
   - `fit` 중심이 아니라 `contain` 또는 letterbox 전략으로 변경
   - 주요 비주얼이 잘리지 않는 것을 우선

2. 카드 프레임 구조 강화
   - 카드 바깥 프레임, 내부 그리드, 정보 패널, hero 패널의 계층을 더 분명하게 나눈다
   - 보고서 문법에 맞는 경계/배경/간격 체계를 설계한다

3. 이미지-텍스트 분리
   - 현재처럼 "모든 시각 구조를 하나의 PNG로 굽는" 구조를 줄인다
   - 가능하면 HTML/CSS에서 카드 프레임과 텍스트를 담당하고, raw visual은 별도 이미지로 넣는다

4. 목표 품질 기준 상향
   - 목표는 "오른쪽 그림이 들어간 카드"가 아니라 "회의 내용을 설명하는 보고서형 인포그래픽"이다
   - 최소 기준:
     - 한국어 텍스트가 실제 정보로 기능할 것
     - 텍스트와 그림이 한 문서 언어로 묶여 보일 것
     - 시선 흐름이 명확할 것
     - generic한 AI 카드보다 회의 주제를 해설하는 자료처럼 보일 것
   - 즉 현재 결과물보다, 사용자가 예시로 보여준 고품질 AI 인포그래픽에 더 가까운 수준을 목표선으로 잡는다

### 예상 변경 파일

- `src/local_meeting_ai_runtime/ai_client.py`
- `src/local_meeting_ai_runtime/html_pdf_renderer.py`
- `doc/templates/html-meeting-output/report.css`

### 완료 기준

- 오른쪽 비주얼이 의미 없이 잘리지 않는다.
- 카드가 문서 안에서 툭 튀지 않고, 보고서 시각자료처럼 보인다.
- 이미지가 본문과 시각적으로 더 자연스럽게 연결된다.
- 시각자료가 "템플릿 카드"가 아니라 "회의 내용을 설명하는 인포그래픽"처럼 읽힌다.

## 4-4. 축 4: 내부 생성 지시문 노출 문제

### 증상

- 결과물 본문에
  - "핵심 논의 주제와 직접 연결되는 ..."
  - "화이트 베이스, 절제된 네이버 그린 ..."
  같은 내부 생성 지시문이 그대로 노출됨

### 추정 원인

- `postprocess_requests`의 `instruction` 또는 내부 briefing 문장이 최종 렌더링 블록에 그대로 노출되고 있음
- "시각자료 설명"과 "시각자료 생성 프롬프트"가 분리되지 않았음

### 패치 방향

1. 공개용 캡션과 내부 프롬프트 분리
   - 내부 필드:
     - `instruction`
     - `placement_notes`
     - 생성용 style prompt
   - 공개 필드:
     - `title`
     - `caption`
     - 선택적 짧은 해설

2. 최종 렌더링 whitelist 도입
   - 최종 PDF에 출력 가능한 필드만 명시적으로 렌더링
   - 내부 생성용 필드는 HTML/PDF에 절대 노출하지 않음

3. 회귀 테스트 추가
   - 최종 HTML/PDF 텍스트 안에
     - `생성한다`
     - `화이트 베이스`
     - `절제된`
     같은 생성 지시문 패턴이 남지 않도록 검사

### 예상 변경 파일

- `src/local_meeting_ai_runtime/html_pdf_renderer.py`
- `src/local_meeting_ai_runtime/summary_pipeline.py`
- 테스트 파일 추가/보강

### 완료 기준

- 내부 생성 프롬프트 문장이 최종 PDF/HTML에 더 이상 보이지 않는다.
- 시각자료 아래에는 사용자에게 보여도 되는 제목/캡션만 남는다.

---

## 4-5. 축 5: 추상적 폰트 지시를 실제 폰트 선택으로 연결하지 못하는 문제

### 증상

- 사용자가 구체 폰트명을 주지 않으면 결과물이 결국 `Malgun Gothic` fallback으로 수렴함
- "둥글둥글한", "정돈된", "부드럽지만 장난스럽지 않은", "브랜드 느낌에 맞는" 같은 추상적 지시가 실제 폰트 선택으로 이어지지 않음
- 문서 본문 폰트와 시각자료 내부 폰트가 같은 fallback으로 수렴해, 시각자료의 무드와 타이포그래피가 충돌함
- 특히 회의록 본문은 보수적인 기본 고딕이라도 봐줄 수 있지만, 둥글고 부드러운 시각자료 안에 같은 딱딱한 기본 폰트가 들어가면 품질이 급격히 떨어져 보임

### 추정 원인

- `renderer_title_font / renderer_heading_font / renderer_body_font`가 비어 있으면 렌더러가 바로 fallback을 씀
- 폰트 결정층이 "명시적 이름이 있으면 적용" 수준에 머물러 있음
- 문서용 폰트와 시각자료용 폰트를 별도 계층으로 다루지 않고 있음

### 패치 방향

1. 폰트 선택 전략 계층 추가
   - 입력:
     - SKILL 원문
     - 회사/브랜드 조사 결과
     - 문서 톤
   - 출력:
     - 실제 title/heading/body font family

2. 추상 표현 -> 폰트 패밀리 매핑
   - 예:
     - 둥글둥글함
     - 차분한 보고서
     - 기술 브리핑
     - 친근하지만 과하지 않음
   - 단, 과거처럼 단순 키워드 카운팅이 아니라 문맥 전체를 보고 결정

3. 문서 타이포와 시각자료 타이포 분리
   - 문서용:
     - `document_title_font`
     - `document_heading_font`
     - `document_body_font`
   - 시각자료용:
     - `visual_title_font`
     - `visual_label_font`
     - `visual_body_font`
   - 문서 본문은 회의록 가독성 중심의 보수적 선택을 유지할 수 있지만,
     시각자료 내부 텍스트는 카드 무드와 어울리는 별도 폰트를 고를 수 있어야 한다
   - 즉 "본문은 안정적, 시각자료는 더 부드럽고 어울리게" 같은 분리가 가능해야 한다

4. HTML-first 폰트 자산 전략 확정
   - 시스템 설치 폰트만 의존하지 않음
   - 필요하면 `assets/fonts/` 번들 + `@font-face` 사용
   - 라이선스 확인 가능한 폰트만 포함

### 예상 변경 파일

- `src/local_meeting_ai_runtime/ai_client.py`
- `src/local_meeting_ai_runtime/html_pdf_renderer.py`
- `doc/templates/html-meeting-output/report.css`
- 새 폰트 자산 폴더(필요 시)

### 완료 기준

- 사용자가 구체 폰트명을 안 줘도, 추상적 무드 지시가 실제 폰트 선택으로 이어진다.
- 브랜드/무드에 따라 결과물 폰트 인상이 달라진다.
- 아무 단서가 없을 때만 마지막 fallback으로 기본 폰트로 간다.
- 문서 본문 폰트와 시각자료 내부 폰트는 필요 시 서로 다르게 선택될 수 있다.
- 시각자료 안의 타이포그래피가 카드의 둥글고 부드러운 무드와 충돌하지 않는다.

## 4-6. 축 6: 시각자료가 사실상 단일 템플릿 반복처럼 보이는 문제

### 증상

- 시각자료마다 오른쪽 그림만 다르고, 나머지 형식은 거의 동일함
- 제목 + 핵심 포인트 3개 + 오른쪽 그림 구조가 반복됨
- 사용자 입장에서는 "자유롭게 구성된 시각자료"보다 "같은 카드 템플릿 반복"으로 보임
- 고품질 AI 인포그래픽 예시와 비교하면, 현재 결과물은 구조 다양성, 정보 밀도, 시각 리듬이 모두 부족해 보임

### 추정 원인

- 현재 `_compose_result_visual_card(...)`가 고정된 캔버스/고정된 양분 구조를 사용
- `points[:3]`로 항상 3개 포인트 박스를 만든다
- 상단 밴드 / 좌측 패널 / 우측 hero 구조가 모든 카드에 공통

### 패치 방향

1. 카드 layout variant 도입
   - 최소 3가지 이상
     - split report card
     - full-width diagram card
     - compact side note card
     - timeline / matrix / comparison variant

2. 섹션 의미에 따라 variant 선택
   - 비교형 주제 -> matrix/comparison
   - 흐름 설명 -> timeline/flow
   - 구조 설명 -> diagram/full-width
   - 핵심 bullet 요약 -> compact report card

3. "핵심 포인트 3개" 고정 제거
   - 포인트 수는 1~N 유동
   - 어떤 카드는 포인트 없이 이미지+짧은 설명만
   - 어떤 카드는 2열 비교
   - 어떤 카드는 작은 요약 칩만

4. 텍스트와 비주얼을 HTML이 조립
   - PNG 하나에 모든 정보를 굽지 않는다
   - raw image는 raw image대로 두고
   - 카드 구조는 HTML/CSS가 조립하도록 옮긴다

5. 목표 예시 대비 품질 격차를 명시적 기준으로 관리
   - 사용자가 보여준 예시처럼:
     - 제목이 강한 정보 구조를 만들고
     - 한국어 설명 텍스트가 실제 주제 해설로 기능하고
     - 도식/카드/비교 구조가 한 화면 안에서 다양하게 엮이며
     - 이미지 자체가 generic 장식이 아니라 논지 설명을 돕는 수준
     까지를 목표 품질 기준으로 둔다
   - 즉 "현재보다 조금 나아짐"이 아니라, "예시와 비교해도 납득 가능한 인포그래픽 품질"을 목표로 본다

### 예상 변경 파일

- `src/local_meeting_ai_runtime/ai_client.py`
- `src/local_meeting_ai_runtime/html_pdf_renderer.py`
- `doc/templates/html-meeting-output/report.css`
- 필요 시 카드 variant 템플릿 파일

### 완료 기준

- 모든 시각자료가 같은 형식으로 반복된다는 느낌이 줄어든다.
- 주제별로 다른 카드 문법이 보인다.
- "오른쪽 그림만 바뀐 같은 카드"라는 인상이 사라진다.
- 시각자료 품질이 사용자가 제시한 고품질 AI 인포그래픽 예시와 비교해도 현저히 뒤처진다는 느낌을 줄인다.

## 5. 우선순위 제안

이번 패치는 다음 순서로 가는 것이 가장 안전하다.

### 1차

- 축 1: PDF 인쇄 잡음 및 표지 밀도
- 축 2: 페이지 흐름과 과도한 여백
- 축 4: 내부 생성 지시문 노출 제거

이 세 개는 지금 PDF를 보는 순간 바로 거슬리는 문제이므로 체감 개선폭이 가장 크다.

### 2차

- 축 3: 이미지 잘림/프레임/경계
- 축 6: 시각자료 단일 템플릿 반복 해소

이 두 개는 시각자료 품질 자체를 끌어올리는 구조 작업이다.

### 3차

- 축 5: 추상적 폰트 지시 -> 실제 폰트 선택

이건 품질적으로 매우 중요하지만, 폰트 자산 전략과 같이 가야 해서 약간 더 설계형 작업이다.

## 6. 검증 방식

패치 완료 판단은 테스트 통과만으로 하지 않는다.  
반드시 실제 결과물 PDF 기준으로 본다.

### 코드 검증

- HTML 렌더러 단위 테스트
- artifact export 통합 테스트
- 내부 지시문 누출 회귀 테스트
- 카드 variant 선택 테스트

### 산출물 검증

동일 세션 산출물로 재렌더링하여 확인:

1. 1페이지에 인쇄 헤더/푸터 흔적이 없는지
2. 2페이지 하단이 통째로 비지 않는지
3. 이미지가 잘리지 않는지
4. 생성 지시문이 결과물에 보이지 않는지
5. 추상적 폰트 요구가 실제 결과물 타이포에 반영되는지
6. 시각자료 형식이 서로 다르게 느껴지는지

---

## 7. 최종 목적

이번 추가 패치의 목적은 단순히 HTML-first 프로토타입을 "돌아가게" 만드는 것이 아니다.

최종 목적은 다음과 같다.

- PDF가 진짜 메인 결과물답게 보이게 만들 것
- SKILL의 자유로움이 결과물에 더 자연스럽고 고급스럽게 반영되게 할 것
- 이미지, 폰트, 여백, 표지, 카드 구조가 서로 충돌하지 않고 하나의 문서 언어로 보이게 할 것

즉, 이번 패치의 목표는  
**"HTML-first로 바꿨다"가 아니라, "HTML-first 덕분에 결과물 품질이 눈에 띄게 좋아졌다"를 실제 PDF로 증명하는 것**이다.

## 8. 2026-04-12 즉시 반영 메모

- _compose_result_visual_card(...) 중심의 카드/표 합성 경로는 폐기 대상으로 본다.
- 앞으로의 시각자료 기본 원칙은 완성형 생성 이미지 1장이다.
- 시스템은 생성된 이미지를 다시 왼쪽 패널/포인트 3개/캡션 카드로 재조립하지 않는다.
- HTML/PDF는 시각자료를 배치만 하며, 이미지 자체를 표처럼 다시 만드는 일을 하지 않는다.
- 기존 세션에 남아 있는 *-card-*.png 경로는 호환성 때문에 읽을 수는 있지만, raw generated image가 있으면 그것을 우선한다.
