# Meeting Output Design Agent System Direction

## 1. 왜 필요한가

결과물 생성 시스템에서 내용 정리와 요약만으로는 충분하지 않다.  
사용자는 종종 결과물의 디자인 방향까지 함께 요청한다.

예를 들면 다음과 같다.

- 카카오 느낌이 나게
- 푸근하고 몽글몽글하게
- 딱딱한 브리핑 문서처럼
- 브랜드 느낌은 빼고 담백하게
- 젊고 실험적이되 문서는 정돈되게

문제는 이런 요청이 지금까지는 자주 "색상 변경" 정도로 축소될 수 있었다는 점이다.  
이 상태는 사용자의 창의성을 열어두는 구조가 아니라, 사실상 같은 틀을 유지한 채 표면만 조금 바꾸는 구조에 가깝다.

그래서 결과물 생성 시스템 내부에는 별도의 디자인 담당 층, 즉 `디자인 담당 요원`이 필요하다.

## 2. 핵심 원칙

디자인 담당 요원의 핵심 원칙은 아래와 같다.

1. 사용자의 디자인 의도를 가볍게 취급하지 않는다.
2. 회사/브랜드가 언급되면 기억으로 추정하지 않고 외부 조사를 먼저 한다.
3. 회사가 언급되지 않아도 추상적인 디자인 요청을 해석해야 한다.
4. 디자인 요청을 단순 색상 변경으로 축소하지 않는다.
5. 사용자의 자유를 열어두되, 고정된 미적 취향을 강요하지 않는다.
6. 사용자의 요청을 내부 디자인 문법으로 번역하고 실제 결과물 레이아웃 변화로 이어지게 한다.

## 3. 디자인 담당 요원의 역할

디자인 담당 요원은 "예쁘게 꾸미는 기능"이 아니다.  
이 요원은 사용자의 디자인 의도가 결과물에서 무시되지 않도록 끝까지 책임지는 역할이다.

구체적인 역할은 다음과 같다.

1. 사용자 요청에서 디자인 의도 추출
2. 회사/브랜드/공개 시각 정체성 관련 단서 수집
3. 추상적인 무드 표현 해석
4. 문서 스타일 전략 수립
5. 내부 렌더링 토큰으로 변환
6. 실제 결과물 반영 여부 확인

## 4. 입력

디자인 담당 요원은 다음 입력을 본다.

- 사용자 자연어 요청
- active meeting-output skill
- 결과물 기본 0 상태
- 회사명/브랜드명/산업군/대상 청중 정보
- 공식 사이트, 공식 가이드, 공개 브랜드 단서
- 회의의 성격
  - 브리핑
  - 제안
  - 전략 논의
  - 강의형 회의
  - 실무 보고

## 5. 출력

디자인 담당 요원은 사용자에게 내부 키를 노출하는 대신, 시스템 내부에서 다음과 같은 결과를 만든다.

### 5-1. 디자인 해석 결과

- 브랜드/무드 해석
- 문서 톤
- 문서 태도
  - 친근함
  - 절제됨
  - 실험성
  - 정제됨
  - 공공기관형
  - 제안서형
  - 브리핑형

### 5-2. 레이아웃 전략

- 표지 정렬 방식
- kicker 또는 badge 필요 여부
- 배경 tint 필요 여부
- 섹션 헤더 처리 방식
- 목차 분위기
- 여백 밀도
- 문단 호흡
- 표/박스/라벨 사용 톤

### 5-3. 렌더링 정책 토큰

현재 1차적으로 다음과 같은 디자인 토큰 계층을 둘 수 있다.

- `renderer_layout_style`
- `renderer_heading_style`
- `renderer_cover_align`
- `renderer_density`
- `renderer_surface_tint_color`
- `renderer_cover_kicker`
- `renderer_primary_color`
- `renderer_accent_color`
- `renderer_neutral_color`
- `renderer_title_font`
- `renderer_heading_font`
- `renderer_body_font`

중요한 점은, 이 토큰들이 최종 목표가 아니라는 것이다.  
이 토큰들은 사용자의 자연어 디자인 의도를 실제 결과물 구조로 옮기기 위한 내부 문법이다.

## 6. 회사/브랜드가 언급된 경우

회사나 브랜드가 언급되면 디자인 담당 요원은 반드시 외부 조사를 먼저 수행해야 한다.

순서는 아래와 같다.

1. 공식 사이트와 공식 디자인/브랜드 관련 자료를 찾는다.
2. 공개적으로 확인 가능한 색상, 타이포그래피, UI 무드, 문서 톤 단서를 모은다.
3. 그 단서를 그대로 복붙하지 않고, 회의 결과물 문서에 맞는 형태로 번역한다.
4. 사용자가 별도로 추가한 무드 요청과 조합한다.

예를 들어 사용자가 이렇게 말할 수 있다.

- 카카오 느낌이 나게
- 카카오 느낌이지만 너무 캐주얼하진 않게
- 카카오와 미팅하지만 공공기관 제출용처럼 정돈되게
- 카카오와 미팅하지만 브랜드 느낌은 굳이 안 넣어도 됨

즉 `회사 느낌`과 `문서 무드`는 항상 분리해서 해석할 수 있어야 한다.

## 7. 회사가 언급되지 않은 경우

사용자는 회사명 없이도 디자인만 요청할 수 있다.

예를 들면 다음과 같다.

- 푸근하고 몽글몽글하게
- 딱딱한 브리핑 문서처럼
- 담백하고 차분하게
- 젊고 실험적으로
- 정보량은 많지만 답답하지 않게

이 경우 디자인 담당 요원은 추상적인 표현을 내부 스타일 문법으로 번역해야 한다.

예시:

- `몽글몽글하게` -> soft layout, airy density, tinted surfaces, rounded kicker/badge 계열
- `딱딱한 브리핑 문서처럼` -> structured layout, compact density, plain/rule heading, 정제된 대비
- `담백하게` -> classic layout, plain heading, muted palette, 최소 장식

단, 이것도 고정 사전처럼 하드코딩된 정답으로 굳히면 안 된다.  
이 해석은 기본 방향일 뿐이고, 항상 사용자의 추가 요구에 의해 달라질 수 있어야 한다.

## 8. 절대 해서는 안 되는 것

다음은 금지 대상이다.

1. 회사명을 보면 하드코딩된 preset을 바로 적용하는 것
2. 디자인 요청을 색상 변경 정도로 축소하는 것
3. 사용자의 추상적 표현을 무시하고 기존 템플릿을 그대로 유지하는 것
4. 내부 키나 구조를 사용자에게 강요하는 것
5. 사용자의 자유를 막는 고정 규칙을 과도하게 늘리는 것
6. "회사 느낌" 하나만 보고 사용자의 문서 무드 요청을 덮어쓰는 것

## 9. 시스템 안에서의 위치

디자인 담당 요원은 다음 흐름 사이에 존재하는 것이 자연스럽다.

1. 사용자 요청 / skill 입력
2. 디자인 의도 해석
3. 외부 조사 또는 무드 해석
4. 디자인 전략 수립
5. 렌더링 정책 토큰 생성
6. summary pipeline 정규화
7. artifact exporter 반영
8. 결과물 검토

즉 skill만으로 끝나지 않는다.  
skill은 사용자의 의도를 전달하는 입구이고, 디자인 담당 요원은 그 의도를 실제 결과물 구조로 번역하는 내부 책임 층이다.

## 10. 디자인 관련 도구 / MCP에 대한 판단

결론부터 말하면, 필수 코어와 품질 향상 도구를 나눠서 봐야 한다.

### 10-1. 필수 코어

초기 버전에서 반드시 필요한 것은 다음이다.

1. 외부 웹 검색
2. 공식 사이트/공개 자료 확인
3. 내부 디자인 토큰 해석기
4. 렌더러 반영 경로

즉 `외부 조사 -> 해석 -> 렌더링`의 기본 루프가 먼저 완성되어야 한다.

### 10-2. 있으면 퀄리티가 올라가는 도구 / MCP

다음은 디자인 담당 요원의 품질을 올려주는 보강 수단이다.

1. 브라우저 MCP
   - 공식 사이트를 열어 실제 시각 톤을 본다.
   - 텍스트 설명만이 아니라 정렬, 여백, 배경, 버튼/배지 톤을 직접 확인한다.

2. 이미지/PDF 비교 도구
   - 레퍼런스 PDF와 현재 결과물을 나란히 보고 차이를 판단한다.
   - 디자인 반영이 실제로 되었는지 검증하는 데 유용하다.

3. 색상/스타일 추출 보조 도구
   - 공개 브랜드 페이지나 레퍼런스 문서에서 메인/보조 색과 표면 톤을 추출한다.
   - 단, 자동 추출 결과를 그대로 믿지 말고 사람이 읽을 수 있는 의미로 재해석해야 한다.

4. 시각 검증 도구
   - 표지, 목차, 섹션 헤더의 배치가 실제로 다르게 보이는지 확인한다.

5. 필요 시 Figma 계열 연동 또는 디자인 자산 MCP
   - 추후 고급 단계에서 유용할 수 있다.
   - 하지만 초기 필수는 아니다.

즉, 디자인 관련 MCP는 "없으면 시작 못 하는 것"보다는 "요원의 안목과 검증력을 높여주는 증폭기"에 가깝다.

## 11. 현재 구현 방향

현재 시스템은 이미 1차 디자인 토큰 계층을 수용하기 시작했다.  
하지만 이것은 시작점이지 완성형이 아니다.

앞으로 필요한 확장은 아래와 같다.

1. 표지 전용 레이아웃 전략 강화
2. 목차 스타일 변형 강화
3. section header / band / badge 표현 강화
4. soft / structured / classic 스타일 차이를 더 크게 확대
5. 디자인 반영 검증 루프 추가
6. 사용자 추상 요청을 더 잘 해석하는 디자인 프롬프트 강화

## 12. 최종 정리

디자인 담당 요원은 선택 기능이 아니다.  
결과물 디자인을 진짜로 열어두려면 거의 필수에 가깝다.

이 요원이 있어야 시스템은 다음처럼 행동할 수 있다.

- "이 회사는 어떤 공개된 느낌을 가지는가?"
- "사용자가 원하는 문서 무드는 무엇인가?"
- "이 요청을 문서 구조와 시각 언어로 어떻게 번역할 것인가?"
- "결과물에 실제로 반영되었는가?"

결국 이 요원의 목적은 하나다.

`사용자의 무궁무진한 디자인 의도를, 시스템이 대충 흘리지 않고 실제 결과물로 책임 있게 번역하게 만드는 것`
