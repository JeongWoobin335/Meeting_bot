# PDF-First 결과물 생성 시스템 방향 정리

작성일: 2026-04-11  
대상 워크스페이스: `C:\Users\jung\Desktop\ZOOM_MEETING_BOT`

## 1. 이 문서를 쓰는 이유

이 문서는 현재 `ZOOM_MEETING_BOT`의 결과물 생성 시스템을

- `DOCX 중심 -> PDF 변환`

구조에서

- `HTML/CSS 중심 -> PDF 직접 렌더링`

구조로 재정의해야 하는 이유와 원칙을 정리하기 위한 설계 문서다.

핵심 목적은 단순히 렌더링 도구를 바꾸는 것이 아니다.  
핵심 목적은 **사용자의 SKILL 지시사항을 더 자유롭고 더 높은 품질로 PDF 결과물에 반영할 수 있는 구조로 바꾸는 것**이다.

즉 이 문서는 `출력 형식 변경` 문서이면서 동시에 `결과물 생성 철학 고정 문서`다.

---

## 2. 절대 잊으면 안 되는 핵심 원칙

### 2-1. 우리가 여는 것은 오직 `결과물 생성` 파트다

이 프로젝트에서 열어두는 것은 결과물 생성 레이어다.

열어두는 영역:

- 제목
- 회의 전체 요약
- 핵심 논의 주제
- 결정사항 / 액션 아이템 / 열린 질문의 표현 방식
- 문서 디자인
- 폰트 / 색 / 표면 처리
- 이미지 생성과 배치
- 페이지 구성과 밀도
- 브랜드 / 분위기 반영

열어두지 않는 영역:

- Zoom 참가
- 음성 수집
- 전사
- 화자 추정
- 참가자 추적
- 세션 관리
- finalization
- 전달 파이프라인
- 내부 schema

즉, **본체 엔진은 유지하고 결과물 생성만 자유롭게 연다**가 계속 핵심이다.

### 2-2. `제기자 / 주요 화자 / 타임스탬프`는 결과물 취향 옵션이 아니다

이 세 가지는 시스템의 핵심 추적 정보다.

- 누가 먼저 말을 꺼냈는가
- 누가 이 논의를 주도했는가
- 어느 시점의 근거인가

이 정보는 결과물의 장식이 아니라 회의 기록의 책임 구조다.  
그러므로 이 값들은 **SKILL로 숨기거나 의미를 바꾸는 대상이 아니다.**

### 2-3. SKILL은 참고문이 아니라 지시사항이다

결과물 생성 파트 안에서 SKILL은

- 한 글자도 빠짐없이
- 최우선 0순위로
- 실제 결과물에 집행되어야 한다

SKILL에 없는 것만 기본 레퍼런스로 간다.  
SKILL에 적힌 것을 시스템이 줄이거나, 편한 내부 메뉴 몇 개로 축소하거나, “비슷한 것”으로 대충 대체하면 안 된다.

### 2-4. `0의 상태`는 시작점이지 감옥이 아니다

기본 레퍼런스는 필요하다.

- 아무 skill이 없어도 안정된 결과물을 만들기 위한 최소 기준

하지만 이 기본 레퍼런스가 사용자를 가두면 안 된다.

- 페이지 수 고정
- 문장 수 고정
- 색상만 바꾸는 가짜 커스터마이징
- 정해진 레이아웃 메뉴 안에서만 고르는 구조

이런 것은 모두 `0의 상태`를 감옥으로 만드는 방식이다.

## 3. 왜 지금 구조가 문제인가

현재 시스템의 출력 흐름은 사실상 다음과 같다.

1. 회의 내용을 요약/정리해 DOCX를 만든다
2. DOCX를 LibreOffice로 PDF로 변환한다

즉 구조상 중심은 아직 `DOCX`다.  
하지만 사용자에게 가장 중요한 최종 산출물은 `PDF`다.

이 불일치 때문에 지금까지 확인된 문제가 발생한다.

### 3-1. 폰트 문제

실제 사례에서 DOCX는 `Malgun Gothic`이 적용되었지만, PDF에서는 폰트가 치환되었다.

즉 현재 구조는

- DOCX에서는 원하는 폰트를 반영해도
- PDF 변환에서 그 폰트를 그대로 유지하지 못할 수 있다

결과적으로 사용자가 보는 최종 문서에서는

- 글자가 원시적으로 보이거나
- 한 문장 안에서 폰트가 섞이거나
- 브랜드 / 분위기와 맞지 않는 인상

이 생긴다.

### 3-2. 여백 문제

이미지 카드가 DOCX 안에서는 큰 표/블록으로 삽입되고, PDF 변환 시 페이지 하단 공간이 애매하면 그 블록이 다음 페이지로 넘어가 버린다.

그 결과:

- 이전 페이지 하단이 크게 비고
- 다음 페이지 맨 위에 카드가 갑자기 등장하고
- 문서 흐름이 끊긴다

이건 내용이 부족해서 생긴 여백이 아니라, **DOCX 레이아웃 블록이 PDF에서 비효율적으로 해석되면서 생긴 여백**이다.

### 3-3. 이미지/카드 품질 문제

현재 시각자료 카드는

- 문서 안에서 따로 논다
- 오른쪽 hero image가 잘리기도 한다
- 카드 경계와 그리드가 약하다
- 폰트와 표면 문법이 너무 기본형이다

이 역시 핵심적으로는 Word 문법을 통해 문서 안에 넣는 방식의 한계와 연결된다.

즉 지금 겪는 세 문제는 별개의 버그가 아니라, 사실상 하나의 구조 문제다.

> **PDF가 메인인데도 DOCX를 기준으로 먼저 만들고 있기 때문에 최종 품질 병목이 계속 생긴다.**

## 4. 방향 전환의 핵심 진단

우리가 바꾸려는 것은 시스템의 본질이 아니다.

바꾸지 않는 것:

- 음성 수집
- 전사
- 참가자 추적
- 제기자 / 주요 화자 / 타임스탬프 복원
- 회의 전체 기록 축적
- briefing JSON 생성
- SKILL 반영 구조

바꾸는 것:

- 결과물 렌더링 기준점
- 문서 출력 문법
- PDF 생성 방식

즉 앞으로의 방향은 다음 한 문장으로 정리된다.

> **시스템 본체는 유지하고, 결과물 생성 방식만 HTML/CSS 기반 PDF 중심으로 바꾼다.**

---

## 5. 왜 HTML/CSS 중심이 더 맞는가

### 5-1. PDF가 메인 결과물이기 때문이다

사용자가 실제로 보는 것은 PDF다.

- 전달도 PDF
- 검토도 PDF
- 인상도 PDF
- 브랜드 느낌도 PDF
- 여백/타이포/이미지 품질도 PDF

그렇다면 설계 기준도 PDF여야 한다.

### 5-2. HTML/CSS는 문서 디자인 자유도가 더 높다

HTML/CSS 기반 PDF는 다음을 더 잘한다.

- 여백 제어
- 카드 / 섹션 / 표지 레이아웃
- 표 / 그리드 구성
- 이미지와 본문 통합 배치
- 페이지마다 반복되는 시각 문법
- 브랜드별 표지/헤더/배지 처리
- 카드형 시각자료와 문단 흐름의 결합

즉 “보고서형”, “브리핑형”, “회사 분위기형” 같은 변형을 구조적으로 더 잘 받는다.

### 5-3. 폰트 자유도가 훨씬 크다

HTML/CSS는

- `@font-face`
- 번들된 폰트 파일
- 프로젝트 내부 assets/fonts

같은 경로로 폰트를 직접 다룰 수 있다.

즉:

- 로컬에 설치된 시스템 폰트만 쓰지 않아도 된다
- PDF를 최종 목표로 브랜드 분위기에 가까운 폰트를 직접 묶을 수 있다
- 폰트 치환 리스크를 크게 줄일 수 있다

이건 현재 DOCX-first 구조보다 훨씬 유리하다.

### 5-4. 이미지 통합이 자연스럽다

HTML/CSS에서는 이미지를

- 섹션 안 그리드의 일부로 넣거나
- 좌우 2단 레이아웃으로 넣거나
- 페이지 전환 포인트에 맞춰 넣거나
- 카드와 캡션을 하나의 구조 안에서 제어

할 수 있다.

즉 지금처럼

- 이미지가 툭 튀어나오고
- 별도 큰 테이블 블록으로 들어가고
- 페이지 하단에 애매하게 걸리다가 밀려버리는

문제를 줄일 수 있다.

---

## 6. 새 구조에서 유지해야 하는 시스템 중심축

HTML/CSS 기반으로 바꾸더라도 아래는 절대 유지해야 한다.

### 6-1. 회의 전체 기록 축적

시스템은 회의 중 발생한 내용을 처음부터 끝까지 누적해 쌓아둔다.

- transcript
- chat
- interaction timeline
- archived audio 기반 최종 전사

즉 재료는 전체로 쌓아두고, 결과물 단계에서 필요한 것을 꺼내 쓰는 구조를 유지해야 한다.

### 6-2. JSON/briefing 중심 데이터 계약

렌더러가 바뀌어도 중심 데이터는 여전히 briefing JSON이어야 한다.

예를 들면:

- title
- executive_summary
- sections
- decisions
- action_items
- open_questions
- postprocess_requests
- participants
- trace information

이 데이터 계약은 유지하고, **DOCX 대신 HTML 렌더러가 그 데이터를 읽게 한다.**

### 6-3. trace fields는 core에서 보장

아래 값들은 여전히 core에서 보장되어야 한다.

- raised_by
- speakers
- timestamp_refs

렌더러는 이 값을 받아 보여줄 뿐, 없애거나 의미를 바꾸는 주체가 아니다.

### 6-4. SKILL은 결과물 레이어에서만 집행

SKILL은 여전히 결과물 생성 레이어만 건드린다.

가능한 것:

- 어떤 블록을 보여줄지
- 어떤 톤으로 쓸지
- 브랜드/무드/폰트/색 방향
- 이미지 개수와 위치
- 페이지 밀도

불가능한 것:

- 음성 수집 방식 변경
- 전사 엔진 변경
- 화자 추적 방식 변경
- 세션 흐름 변경

---

## 7. 새 PDF-first 구조의 기본 아키텍처

권장 구조는 다음과 같다.

1. 회의 종료
2. 기존 엔진이 briefing JSON 생성
3. active SKILL과 기본 레퍼런스를 함께 반영해 결과물 지시를 확정
4. postprocess 이미지가 필요하면 먼저 생성
5. briefing JSON + SKILL + 이미지 자산을 가지고 HTML 문서 생성
6. HTML/CSS를 브라우저 엔진으로 PDF 렌더링
7. 필요하면 별도 보조 산출물로 DOCX 생성

중요한 점:

- 중심은 `HTML -> PDF`
- DOCX는 선택적 부가 산출물

이다.

---

## 8. HTML 렌더러가 가져야 하는 원칙

### 8-1. SKILL 지시사항을 직접 반영

HTML 렌더러는 사용자의 SKILL을 다음에 직접 반영해야 한다.

- 색
- 폰트
- 표지 스타일
- 섹션 구조
- 카드/표 디자인
- 이미지 위치
- 페이지 밀도
- 브랜드 분위기

여기서 다시

- `classic / soft / structured`
- `before / inside / after`

같은 좁은 내부 메뉴로 접으면 안 된다.

### 8-2. 텍스트는 HTML이 책임지고, 이미지는 구조를 책임지게 한다

이미지 품질 문제를 줄이려면:

- raw image model은 구조와 분위기 위주
- 실제 한국어 텍스트는 시스템이 HTML/CSS 안에서 정확히 넣는 방식

이 더 안정적이다.

즉 인포그래픽 전체를 이미지 모델에게 맡기지 않고,

- 구조적 비주얼은 이미지
- 텍스트와 정렬은 HTML/CSS

로 분리하는 것이 바람직하다.

### 8-3. PDF 기준으로 여백과 페이지 흐름을 검토

Word 기준으로 괜찮아 보여도 PDF에서 무너지면 의미가 없다.  
따라서 렌더링 검토 기준은 항상 PDF여야 한다.

점검 대상:

- 폰트가 실제로 원하는 인상인지
- 페이지 하단이 불필요하게 비는지
- 이미지가 잘리는지
- 카드가 본문과 어울리는지
- 회사 / 무드 느낌이 살아 있는지

---

## 9. DOCX의 새로운 위치

이 문서의 방향에서 DOCX는 더 이상 메인 결과물이 아니다.

앞으로 DOCX의 바람직한 위치는 다음 중 하나다.

### 9-1. 보조 산출물

PDF가 메인이고, DOCX는 편집이 필요한 사람을 위한 보조 파일로만 남긴다.

### 9-2. 선택 출력

설정이나 옵션에 따라 DOCX가 필요할 때만 만든다.

### 9-3. 아예 비활성화 가능

사용자가 PDF만 필요하다면 DOCX 생성을 꺼도 된다.

이 방향이 맞는 이유는 분명하다.

> 결과물의 인상과 품질을 결정하는 중심은 PDF이지 DOCX가 아니다.

---

## 10. 예상되는 구현 단계

### 10-1. 1단계: HTML 렌더러 병행 추가

바로 기존 DOCX 경로를 지우기보다,

- 기존 briefing JSON 유지
- 새 HTML renderer 추가
- HTML -> PDF 경로 병행

으로 간다.

### 10-2. 2단계: 네이버 같은 실제 skill로 PDF 비교

동일한 회의 briefing으로

- DOCX-first PDF
- HTML-first PDF

를 나란히 비교해

- 폰트
- 여백
- 이미지 삽입
- 브랜드 느낌

차이를 본다.

### 10-3. 3단계: 이미지 카드 재설계

현재의 고정 카드 템플릿 대신,

- HTML 문서 안에 자연스럽게 녹는 정보 블록
- 잘리지 않는 비주얼 패널
- 더 정교한 타이포와 경계 구조

로 바꾼다.

### 10-4. 4단계: PDF를 기본, DOCX를 보조로 재배치

이 단계에서 시스템 관점도 바꾼다.

- 지금: DOCX 중심 / PDF 파생
- 목표: PDF 중심 / DOCX 보조

## 11. 이 전환에서 절대 금지할 것

다음은 전환 과정에서도 금지한다.

1. 결과물 생성 전환을 핑계로 본체 엔진까지 건드리는 것
2. SKILL을 다시 참고문처럼 취급하는 것
3. HTML 렌더러 안에 새로운 고정 preset 메뉴를 만드는 것
4. PDF 품질을 위해 trace fields를 희생하는 것
5. 디자인을 위해 회의 내용 근거를 약화시키는 것
6. 내부 구조를 사용자에게 그대로 노출하는 것

## 12. 최종 정리

지금까지 본 결과를 종합하면, 현재 품질 병목은 개별 버그 세 개가 아니라 구조 문제다.

- PDF 폰트가 무너진다
- 이미지 때문에 페이지 여백이 과하게 생긴다
- 시각자료 카드가 문서 안에서 따로 논다

이 모든 문제의 출발점은

> `PDF가 메인인데 DOCX를 먼저 만든 뒤 PDF로 바꾸고 있다`

는 점에 있다.

그러므로 앞으로의 방향은 분명하다.

> **시스템의 본질 기능은 유지하고, 결과물 생성 방식은 HTML/CSS 기반 PDF-first 구조로 옮긴다.**

이 전환의 목표는 단순히 기술 스택을 바꾸는 것이 아니라,

- 사용자 SKILL의 자유를 더 매끄럽게
- 더 높은 품질로
- 더 직접적으로
- PDF 결과물에 반영할 수 있게 만드는 것

이다.

즉 이 전환의 본질은 다음 한 문장이다.

> **자유로움을 해치지 않으면서, PDF를 중심에 놓고 결과물 품질을 다시 설계한다.**
