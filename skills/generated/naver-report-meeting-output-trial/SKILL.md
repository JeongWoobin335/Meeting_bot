# 네이버 보고서형 회의 결과물 오버라이드

이 오버라이드는 네이버와 관련된 회의에서 결과물을 화려한 홍보물보다 구조적인 보고서와 회의록에 가깝게 정리하고 싶을 때 쓴다. 문서는 네이버의 공개 브랜드 인상처럼 기술과 서비스의 연결, 견고한 신뢰감, 친근하지만 과장되지 않은 안내자 느낌을 살리되, 전체 톤은 차분하고 실무적인 회의 문서로 유지한다.

## 문서 톤

- 결과물은 홍보 카피나 제안서 과장보다, 내부·외부 공유가 모두 가능한 정돈된 보고서형 회의록처럼 보이게 쓴다.
- 세부 색과 표면 처리는 네이버의 공개 브랜드 리소스가 주는 인상처럼 깨끗한 화이트 베이스, 절제된 네이버 그린 포인트, 견고한 고딕 계열 인상, 기술 기업다운 신뢰감을 우선한다.
- 문서는 서비스, 기술, 운영, 실행 맥락이 잘 읽히는 방향으로 정리하고, 이벤트 홍보물처럼 떠 있는 문구나 과장된 슬로건은 붙이지 않는다.
- 네이버 분위기를 살린다는 이유로 초록색을 과하게 남용하거나, 광고 배너 같은 장식형 레이아웃으로 흐르지 않는다.
- 제목, 헤더, 본문은 모두 가독성이 높은 고딕 계열 인상을 유지하고, 지나치게 둥글거나 장난스러운 폰트 분위기는 피한다.
- 기본 레퍼런스에서 쓰던 기본 폰트를 그대로 반복하지 않는다. 최소한 폰트는 바꾸고, 네이버의 공개 브랜드 인상과 서비스 기업다운 정돈된 분위기를 살리는 방향으로 새로 고른다.
- 문서용 폰트는 먼저 깔끔하고 견고한 네이버 계열 고딕 인상을 우선하고, 실제 결과물의 최종 매체가 PDF라는 점까지 고려해 지나치게 딱딱하거나 반대로 너무 장난스럽지 않은 쪽으로 고른다.
- 꼭 특정 폰트 이름 하나를 고정하려고 하기보다, 네이버의 공개적인 분위기와 이번 회의의 보고서 톤을 가장 잘 살리는 폰트를 우선 선택한다.
- 별도 브랜드 폰트를 직접 쓸 수 없는 환경이라면, 그와 가장 가까운 정돈된 고딕 계열 폰트로 자연스럽게 대체하되 결과물이 기본 레퍼런스와 같은 인상으로 돌아가지는 않게 한다.

## 회의 전체 요약

- `회의 전체 요약`은 기본 상태보다 더 길고 풍부하게 작성한다.
- 단순 압축보다 이번 회의가 왜 중요한지, 어떤 문제의식과 판단축이 있었는지, 무엇이 합의되었고 무엇이 아직 남았는지까지 연결해 드러낸다.
- 읽는 사람이 첫 장에서 회의의 목적, 핵심 흐름, 결과적 의미를 빠르게 이해할 수 있게 만든다.
- 회의 밀도가 높으면 여러 문장으로 길어져도 괜찮고, 페이지 수가 늘어나는 것도 허용한다.

## 핵심 논의 주제

- `핵심 논의 주제`는 시간순 나열보다 논점 축과 아젠다 축으로 재구성한다.
- 각 섹션은 단순 요약에서 끝나지 말고, 논의 포인트, 해석, 제품/서비스 함의, 운영 판단, 실행 조건이 드러나게 충분히 설명한다.
- 네이버 관련 회의라면 검색, 탐색, 서비스 적용, Everyday Tech, On-Service AI, 지속가능성, 사용자 접점, 기술의 일상성 같은 공개적인 회사 맥락을 회의에 실제로 닿는 범위 안에서만 자연스럽게 연결한다.
- 한두 줄로 급히 끊지 말고, 중요한 주제는 길게 풀어도 괜찮다.

## 결정사항 / 액션 아이템 / 열린 질문

- 이 블록들은 회의 근거가 있는 내용만 쓴다.
- 실제로 정리된 내용이 없으면 억지로 채워 넣지 않는다. 비어 있는 블록을 억지 요약으로 부풀리지 않는다.
- 결정사항은 실제로 정리된 판단만 적고, 액션 아이템은 확인 가능한 다음 행동으로 적는다.
- 액션 아이템이 존재한다면 이 블록은 기본 상태보다 더 또렷하게 보여야 한다. 다음 행동, 책임 주체, 확인 포인트, 순서를 읽는 사람이 바로 파악할 수 있게 분명하게 정리한다.
- 열린 질문은 미해결 이슈, 추가 확인 포인트, 검토가 필요한 조건을 드러내는 쪽으로 쓰되, 실제 회의 근거가 없는 generic 질문을 꾸며 넣지 않는다.

## 제외와 주의

- `리스크 신호` 블록은 사용하지 않는다.
- `메모` 블록은 사용하지 않는다.
- 회의와 직접 연결되지 않는 generic stock 이미지, 불필요한 마케팅 문구, 공허한 미래 비전 문장은 줄인다.
- 네이버 로고나 브랜드를 장식 요소처럼 남용하지 않는다. 특히 로고를 라인, 프레임, 반복 패턴처럼 쓰지 않는다.

## 시각 자료

- 핵심 논의 주제마다 최소 1장 이상의 이미지를 만든다.
- 특정 주제가 매우 중요하거나 설명이 복잡하면 이미지를 더 늘려도 된다.
- 이미지는 장식이 아니라 이해 보조여야 하며, 각 이미지가 어느 핵심 논의 주제와 연결되는지 분명히 드러나야 한다.
- 정보형 인포그래픽, 개념 구조도, 서비스-기술 연결 다이어그램, 운영 판단 구조, 실행 로드맵형 시각자료처럼 회의록 읽기에 도움이 되는 방향을 우선한다.
- 이미지 안에 들어가는 한글 텍스트는 세션 주제와 직접 연관되어야 하고, 깨진 글자, 가짜 한글, 의미 없는 문구, 주제와 무관한 문장을 허용하지 않는다.
- 이미지는 최종 PDF 마무리 전에 생성되어 문서 안에 실제로 들어가야 한다.

## 분량 원칙

- 페이지 수를 숫자로 고정하지 않는다.
- 대신 기본 상태보다 더 풍부하고 길게 쓰는 방향을 택한다.
- 회의가 길거나 설명 밀도가 높으면 요약, 핵심 논의 주제, 시각 자료가 자연스럽게 늘어나면서 페이지 수가 많아져도 괜찮다.
