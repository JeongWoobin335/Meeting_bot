# 벨빈 코리아 미팅 결과물 오버라이드

이 오버라이드는 벨빈 코리아와의 회의처럼 팀 역할, 협업, 자기이해, 강점 기반 운영 언어가 자연스럽게 등장하는 회의에서 쓴다. 문서 전체는 과장된 장식보다 차분한 전문성, 구조화된 팀 인사이트, 실제 적용 가능성을 먼저 보여준다.

## 문서 톤

- 결과물은 화려한 홍보물보다 구조적인 브리핑 문서처럼 보이게 쓴다.
- 세부 색, 폰트, 장식은 미리 고정하기보다 벨빈 코리아의 공개적인 회사 분위기와 이번 회의 맥락을 함께 반영해 정한다.
- 회의 내용과 맞으면 팀 역할, 행동 강점, 협업 균형, 자기인식, 상호보완, 역할 충돌 같은 표현을 적극적으로 살린다.
- 회의에 없는 Belbin 개념을 억지로 끼워 넣지는 않는다.

## 회의 전체 요약

- `회의 전체 요약`은 기본 상태보다 더 길고 풍부하게 작성한다.
- 단순 압축보다 이번 회의의 문제의식, 핵심 프레임, 중요한 합의와 차이, 이후 적용 의미까지 연결해 준다.
- 읽는 사람이 첫 장에서 "이 회의가 왜 중요했고 무엇이 남았는지"를 바로 이해할 수 있게 만든다.
- 짧은 회의는 간결하게 유지하되, 개념 밀도나 논의 밀도가 높으면 충분히 길어져도 괜찮다.

## 핵심 논의 주제

- `핵심 논의 주제`는 시간순 나열보다 아젠다 축이나 논점 축으로 재구성한다.
- 각 섹션은 단순 요약에 그치지 말고, 논의 포인트, 해석, 실무 함의가 드러나게 충분히 설명한다.
- 팀 역할 균형, 협업 방식, 강점 활용, 역할 충돌, 실행 설계 같은 주제가 나오면 섹션 안에서 분명히 연결한다.
- 각 섹션에서 `제기자`와 `주요 화자`가 드러날 수 있으면 다시 표시해, 누가 어떤 맥락에서 논의를 이끌었는지 보이게 한다.
- 한두 줄 요약으로 급히 끝내지 말고, 회의 밀도에 맞게 필요한 만큼 자세히 쓴다.

## 검토 사항

- `검토 사항`은 단순한 미해결 질문 목록이 아니다.
- 아직 확정하지 않았지만 비교, 검증, 내부 판단, 추가 확인, 보완 설계가 필요한 항목을 정리하는 블록으로 쓴다.
- 선택지 사이의 판단 포인트나 추가 확인 조건이 있으면 함께 적는다.

## 제외와 주의

- `메모` 블록은 사용하지 않는다.
- `리스크 신호` 블록은 사용하지 않는다.
- 회의와 무관한 일반론, 뻔한 팀워크 격언, 장식용 문장은 줄인다.
- 벨빈 느낌을 흉내 내기 위해 추상적인 영어 슬로건이나 과장된 브랜드 카피를 덧붙이지 않는다.
- 핵심 논의와 직접 연결되지 않는 generic office stock 느낌의 시각 요소는 피한다.

## 시각 자료

- 회의 아젠다나 핵심 논의 주제가 시각화 가치가 있으면, nano-banana 2 계열 이미지 브리프로 핵심 논의 주제와 직접 연결되는 이미지를 최소 1장 이상 만든다.
- 이미지 수를 미리 고정하지는 않는다. 시각화 가치가 높은 핵심 논의 축이 여러 개라면 2장 이상으로 늘려도 된다.
- 이미지는 장식이 아니라 이해 보조여야 하며, 각 이미지가 어느 핵심 논의 주제와 연결되는지 분명히 드러나야 한다.
- 개념 구조, 역할 관계, 협업 장면, 적용 맥락처럼 문서를 읽는 데 도움이 되는 방향으로 프롬프트를 잡는다.
- 이미지는 최종 PDF 마무리 전에 생성되어 문서 안에 실제로 들어가야 한다.

## 분량 원칙

- 이 오버라이드는 페이지 수를 숫자로 고정하지 않는다.
- 대신 기본 상태보다 더 풍부하고 길게 쓰는 방향을 택한다.
- 긴 회의나 설명 밀도가 높은 회의는 요약, 핵심 논의 주제, 시각 자료가 자연스럽게 늘어나면서 페이지 수도 더 많아져도 괜찮다.
