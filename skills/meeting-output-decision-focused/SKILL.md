# 의사결정 중심 회의 결과물 스킬

이 스킬은 `ZOOM_MEETING_BOT`가 회의 결과물을 만들 때, 배경 설명보다 `결정`, `실행`, `남은 쟁점`을 더 선명하게 드러내도록 돕는 예시 스킬입니다.

쉽게 말하면:
- 논의가 길었더라도
- 최종적으로 무엇이 정해졌고
- 무엇이 아직 안 정해졌고
- 누가 다음에 무엇을 해야 하는지를

먼저 보이게 만드는 스타일입니다.

## 언제 쓰면 좋은가

- 프로젝트 진행 회의
- 운영 회의
- 다음 액션이 중요한 내부 정리 회의
- “결론이 뭐였지?”를 빠르게 다시 봐야 하는 회의

## 이 스킬의 핵심 원칙

- 한국어 문어체로 자연스럽게 쓴다.
- 회의에서 실제로 정해진 것과 아직 안 정해진 것을 구분해서 쓴다.
- 배경 설명은 필요하지만, 결정사항보다 앞에 나오지 않게 한다.
- 전사 원문을 그대로 복사하지 말고 정리된 회의록 문장으로 바꾼다.
- 회의 전체 흐름보다 실행에 중요한 내용을 더 우선한다.

## 항목별 작성 방식

### `title`

- 회의의 실제 안건을 짧고 분명하게 드러낸다.
- 가능하면 `무엇을 논의했고 무엇을 정하려 했는지`가 보이게 쓴다.

예:
- `봇 배포 방식과 후속 일정 정리`
- `프로젝트 채널 전달 기준 결정 회의`
- `회의 결과물 스킬 분리 방향 검토`

### `executive_summary`

- 맨 위 `회의 전체 요약`은 2~3문장으로 쓴다.
- 첫 문장에는 회의의 공통 안건을 쓴다.
- 둘째 문장에는 무엇이 정해졌는지 또는 어떤 방향으로 기울었는지를 쓴다.
- 셋째 문장에는 아직 남은 쟁점이나 추가 확인 포인트가 있으면 짧게 쓴다.

### `summary`

- `executive_summary`보다 조금 더 자세히 쓴다.
- 논의 배경을 짧게 적되, 결국 무엇을 하기로 했는지가 흐려지지 않게 한다.
- 중요한 예시가 있으면 한 개 정도는 남긴다.

### `sections`

- 섹션은 3~6개로 나눈다.
- 가능하면 논의 주제보다 `결정 포인트` 중심으로 묶는다.
- 각 섹션 요약은 보통 2~4문장으로 쓴다.
- 어떤 결론이 났는지 먼저 쓰고, 필요하면 이유나 맥락을 뒤에 덧붙인다.
- 결론이 안 난 섹션이라면 “추가 확인이 필요하다”는 점을 분명히 적는다.

### `decisions`

- 실제로 정해진 것만 쓴다.
- 애매한 의견 교환은 넣지 않는다.
- 읽는 사람이 바로 “이번 회의에서 확정된 게 뭐지?”를 알 수 있어야 한다.

### `action_items`

- 가능한 경우 누가, 무엇을, 왜 이어서 해야 하는지가 보이게 쓴다.
- 길게 쓰지 말고 실행 가능한 문장으로 쓴다.

예:
- `우빈이 meeting-output skill 선택 config를 README에 정리한다.`
- `참석자 4인 이상 회의에서 주요 화자 판별을 추가 테스트한다.`

### `open_questions`

- 회의에서 남겨 둔 질문만 적는다.
- 앞으로 무엇을 검증하거나 결정해야 하는지가 보이게 쓴다.

### `risk_signals`

- 일정 지연, 역할 불명확, 운영 복잡성, 설치 부담처럼 실행에 걸림이 되는 요소를 적는다.
- 단순한 분위기나 감상은 넣지 않는다.

## 이 스킬이 추구하는 결과물 느낌

좋은 결과물은:
- 회의를 못 들은 사람도
- “이번에 뭘 하기로 했는지”
- “누가 이어서 뭘 해야 하는지”
- “아직 뭐가 안 끝났는지”

를 빠르게 파악할 수 있어야 합니다.

## 충돌 규칙

- 회의별 운영자 지시가 더 구체적이면, JSON 형식을 깨지 않는 범위에서 운영자 지시를 우선한다.
- 이 스킬과 시스템의 필수 JSON 형식이 충돌하면 시스템 형식을 우선한다.
